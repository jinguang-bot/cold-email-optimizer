# 外贸开发信对抗优化系统（Cold Email Duel）

## 一句话描述

通过买卖双方智能体对抗训练，系统性发现和沉淀高转化外贸开发信策略。

---

## 项目背景

外贸开发信是B2B获客的核心手段之一，但长期以来面临一个根本困境：**没有人能精确定义什么是"好的开发信"，但结果非常容易衡量——对方回复了没有。**

传统优化方式依赖人工经验、小样本A/B测试，迭代慢、成本高、覆盖面窄。本项目试图用对抗模拟的方法，将这个试错过程规模化、自动化。

### 核心洞察

- **结果可定义，过程不可定义** → 这是强化学习的经典适用场景
- **环境可模拟** → 邮件是纯文本交互，LLM角色扮演成本低
- **成功路径可沉淀** → 说服买方的完整邮件链路本身就是可复用的知识资产

---

## 核心方法

### 双智能体对抗框架

```
卖方Agent（Seller）          买方Agent（Buyer）
    │                            │
    ├── 撰写开发信 ──────────────►│
    │                            ├── 判断：拒绝/回复/兴趣
    │◄─────── 买方反馈 ──────────┤
    ├── 根据反馈调整策略          │
    ├── 撰写跟进邮件 ───────────►│
    │                            ├── 判断：拒绝/回复/成交
    │         ...多轮...          │
    ▼                            ▼
   记录对抗路径（成功或失败）
```

### 角色设计

**卖方（Seller Agent）**
- 目标：让买方回复并表现出继续沟通的兴趣
- 需要探索的策略维度：
  - 开场方式（直球/故事/数据/痛点/推荐人）
  - 邮件长度（简短/中等/详细）
  - 价值主张呈现方式
  - CTA（Call to Action）设计
  - 跟进节奏和话术
- 策略多样性保障：通过风格约束强制探索不同策略空间

**买方（Buyer Agent）**
- 目标：除非被充分说服，否则拒绝（基础拒绝率 ~90%）
- 需要模拟的维度：
  - **行业**：制造业、零售、科技、医疗等
  - **职位**：采购经理、CEO、技术总监等
  - **文化**：美式/欧式/中东/东南亚的沟通偏好
  - **痛点**：成本、质量、交付、合规、现有供应商锁定等
  - **拒绝风格**：冷漠无视、礼貌婉拒、直接拒绝、质疑型
- 难度分级：从"容易说服"到"几乎不可能"，形成课程学习梯度

### 奖励信号设计

不仅仅看最终结果，设计多层次奖励：

| 层级 | 信号 | 说明 |
|------|------|------|
| L0 | 邮件被打开 | 买方没有直接忽略（模拟中通过买方是否"读完"体现） |
| L1 | 买方提出疑问 | 从拒绝转为质疑，说明兴趣萌芽 |
| L2 | 买方要求更多信息 | 主动索取 = 明确兴趣信号 |
| L3 | 买方同意继续沟通 | 对抗成功，进入商机培育阶段 |
| L4 | 买方表达购买意向 | 最高级别成功 |

## 项目阶段规划

### Phase 0 — 基础验证（MVP）
- 目标：跑通双Agent对抗的基本pipeline
- 交付：
  - 卖方/买方Agent的prompt设计
  - 单轮+多轮邮件对抗执行引擎
  - 对抗过程记录与存储
- 规模：20-30个买方画像 × 50轮对抗 ≈ 1000-1500个样本
- 评估指标：不同策略的回复率差异是否有统计显著性

### Phase 1 — 策略探索与校准
- 目标：验证模拟结果与真实世界的相关性
- 交付：
  - 策略分类体系（哪些维度真正影响结果）
  - 用真实历史开发信数据做校准
  - 不同行业/文化的策略差异分析
- 关键问题：模拟中"有效"的策略在现实中是否同样有效？

### Phase 2 — 知识沉淀与产品化
- 目标：将成功路径转化为可指导写作Agent的知识库
- 交付：
  - 策略规则库（什么场景用什么策略）
  - 优质示例库（按行业/文化/痛点分类的成功邮件链路）
  - 指导写作Agent的系统prompt或few-shot模板
  - 可选：在线学习闭环（真实发送结果反哺模拟）

## 关键风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| 模拟-现实Gap | LLM买方≠真实买方，可能过度理性或过度礼貌 | 用真实历史数据校准；引入真实邮件做对比实验 |
| 奖励稀疏 | 开发信回复率本就低（1-5%），大量轮次无信号 | 多层次中间奖励；调整买方难度梯度 |
| 策略坍缩 | 卖方收敛到少数固定套路 | 强制策略多样性；定期注入新约束 |
| 买方刻板化 | 所有买方回复风格趋同 | 丰富买方prompt变体；参考真实拒绝邮件语料 |
| 过度优化 | 策略对特定模型版本过拟合 | 多模型交叉验证（如同时用GPT/Claude/GLM跑买方） |
